# v1.1 Roadmap

Features that didn't make v1 but are likely to land before public ship.
Each item has a motivation, a concrete proposal, and a validation gate.

---

## Video brief: agent-authored, video-specific scoring lens

### Problem

Today's scoring is generic. The harness model receives:

- a fixed `rubric_prompt` (hook, shareability, standalone clarity, payoff,
  delivery energy, quotability) plus spike categories,
- per-clip context (transcript window, mining signals, vision signals),
- optional `creator_patterns` learned from past keep/skip feedback.

What it does *not* receive: any synthesis of what makes a clip from
**this specific video** worth keeping. Each clip is scored myopically,
without the model ever seeing the video's overall arc.

This is a meaningful ceiling. Experienced human editors don't score
clips in isolation — they watch the whole video first, identify the
spine, then pick clips that serve that spine. We're asking the model to
do the second half of that work without doing the first.

### Concrete example that motivated this

A test video — *"Why I Stopped Trading Crypto.. [Stream Recap]"*, single-
speaker stream recap — was queued to dogfood `/clip` in Claude Code.
Before running it, the frontend agent (Claude in this session) was asked
what kinds of clips the engine should expect to surface. From the title
alone the agent predicted with high specificity:

1. The central "why I quit" reveal — the thesis moment.
2. Loss/horror anecdotes with specific dollar amounts (the new
   `big_number` spike category from `e0ab071` should fire here).
3. Contrarian takes about crypto culture.
4. Self-deprecating retrospection ("I was an idiot for…").
5. Strong cold-open hooks typical of stream recap edits.

That kind of video-specific anticipation is exactly what the scoring
stage is missing. The agent had it for free from the title; the engine
will discover it (if at all) only implicitly through the rubric.

### Proposal

Insert a new `brief` stage between `mine` and `score` that emits a
small, structured `VideoBrief` written by the harness model from the
full transcript + ingest metadata. The brief is attached to every
subsequent scoring request (and packaging request) as additional
context.

Schema sketch:

```json
{
  "video_id": "<job_id>",
  "rubric_version": "<x.y.z>",
  "theme": "personal turning point — quitting crypto trading",
  "format": "stream recap, single-speaker talking-head",
  "audience": "crypto-curious, finance-burned",
  "expected_viral_patterns": [
    "the central 'why I quit' reveal",
    "specific dollar-amount loss anecdotes",
    "contrarian crypto-culture takes",
    "self-deprecating retrospection"
  ],
  "anti_patterns": [
    "technical TA jargon",
    "stream-chat banter",
    "intro music / dead air"
  ],
  "tone": "candid, retrospective, mildly self-deprecating",
  "notes": "free-form 1-2 sentences"
}
```

The scoring prompt becomes: *"Score this clip against the rubric. The
following `VideoBrief` describes what makes a clip from this video
specifically worth keeping; weight rubric dimensions and spike
categories accordingly."*

### Pipeline integration

1. **New stage** — `brief`. Reads `transcript.json` + ingest metadata,
   writes `brief-request.json` (handoff to harness model) and consumes
   `brief-response.json`. Validates against a Pydantic `VideoBrief`
   model with `extra="forbid"` like every other contract.
2. **Cached per workspace** — re-running scoring does not re-pay the
   brief cost. The brief is invalidated only when the transcript
   changes (different ASR model) or `rubric_version` bumps.
3. **Attached to scoring + packaging** — `scoring-request.json` and
   `package-request.json` gain an optional `video_brief` field. Schema
   `extra="forbid"` does *not* break old payloads because the field is
   added, not required (current consumers ignore it).
4. **Stage ordering** — `mine` still produces clip windows (the brief
   doesn't change which windows we mine); the brief only changes how
   they're scored. Stage `auto` runs `mine -> brief -> score -> review
   -> render`.
5. **Long-transcript handling** — for transcripts above some token
   budget (e.g. 30k chars), summarize chapters or windowed segments
   first; pass the summary into the brief request rather than the raw
   transcript. Same precedent as how packaging handles long context.
6. **Disabling** — `output_profile.video_brief: bool = True` gates the
   stage. Power users can disable per-job to skip the extra handoff.

### Why this is high-leverage

It plays directly to where frontend agents outperform engine
heuristics: synthesizing long context into a compact, opinionated frame.
Right now we use the agent only for per-clip judgement — a narrow role
that a smaller model could plausibly serve. The brief uses the agent
where it actually has an edge over rules and small models, and it does
so once per video (cheap) rather than once per clip (expensive).

It also has a clean upgrade path with `creator_patterns`: high-
confidence creator preferences (e.g. *"this user always cuts intros"*)
become explicit `anti_patterns` in the brief, so the rubric scoring no
longer has to re-derive them from raw signals on every clip.

### Validation gate (when to build)

Run the dogfood `/clip` against the crypto stream recap above on the
current engine. Inspect `review-manifest.json`:

- Did the top clip surface the central "why I quit" reveal?
- Did any loss-anecdote clips with explicit dollar amounts make the
  top 5? Did `big_number` fire on them?
- Did contrarian / self-deprecating moments score competitively?

If the answer to two or more of those is "no" or "only via the
`min_candidates` backfill, not on merit," the generic rubric is
clearly leaving signal on the table and the brief stage is justified
for v1.1.

### Validation gate results (2026-04-25 dogfood)

We ran it. Brief feature is **strongly justified** — and the dogfood
also surfaced a separate, more fundamental miner-stage gap captured
in [miner-quality.md](miner-quality.md). The two compose:
miner-quality.md fixes "what candidates exist," v1.1 brief fixes "how
they get scored."

**The video reality vs. our prediction.** The title — *"Why I Stopped
Trading Crypto.. [Stream Recap]"* — turned out to be loose marketing.
The actual stream is a 28-minute single-host crypto-news monologue
that pivots to a *"shifting attention to AI"* thesis around the
17-minute mark, then brings on a guest trader for the back third
(18:49–26:52) who shills specific stocks (Bloom Energy / $BE, "best
investor of all time" framing). None of the predicted clip patterns
(loss anecdotes, self-deprecation, "why I quit" reveal) actually
exist in the source.

**What the engine produced.** All 5 approved clips were
`min_candidates=5` backfill — every one scored below the 0.70
threshold:

| Clip | Score | Window | Hook |
|------|-------|--------|------|
| clip-002 | **0.58** | 30s | "No drinks, no wine, no nothing — pure focus on AI bottlenecks this weekend." |
| clip-003 | 0.43 | 41s | "Intel just printed $100B in a day — basically all of ETH." |
| clip-001 | 0.42 | 39s | "Ape coin just ripped 140%. He thought BAYC pivoted to AI. They didn't." |
| clip-004 | 0.40 | 42s | "Either you say 'why didn't I sell' or 'why didn't I buy.'" |
| clip-000 | 0.38 | 43s | "Someone burned 75 ETH on HyperLiquid to long a million on ApeCoin" |

**Three concrete findings that justify the brief feature:**

1. **The thesis was right but the model couldn't tell.** The
   highest-scoring clip (clip-002) IS the actual video spine —
   "shifting from crypto to AI." But the model didn't *know* it was
   the spine; it scored 0.58 because the hook is locally strong, not
   because it framed the rest of the video. With a brief saying
   *"theme: shift from crypto trading to AI focus"*, this clip should
   score 0.85+ and the three competing token-pump clips should be
   demoted as off-thesis.

2. **`big_number` spike never fired even on $100B.** Clip-003 leads
   with *"Intel just printed $100B in a day — basically all of ETH"*
   — a textbook `big_number` hook. The model categorized it as
   `unusually_useful_claim` instead. Either the rubric is missing the
   spike trigger or the model wasn't primed to look for it. A brief
   noting *"expected_viral_patterns: ['big numbers from market
   moves']"* would make this explicit.

3. **Redundancy in the selection set.** Clip-001 and clip-000 are
   the same ApeCoin pump from two angles, separated by 3 seconds
   in source time (523–562s vs 560–603s) — they're almost the same
   moment scored twice. Without global context the model has no way
   to recognize duplication. A brief listing the video's distinct
   moments would let scoring deduplicate proactively.

**The bigger finding (not solved by the brief).** The 8-minute guest
interview block — including the explicit Bloom Energy stock pick — was
**never even a candidate**. Mining produced zero windows from
1129–1612s. The brief feature can only re-weight candidates that
mining surfaced; it cannot rescue a window that was never proposed.
That gap is the subject of [miner-quality.md](miner-quality.md), and
it has to land *before* the brief feature to be useful on
interview-format content. If we ship the brief without fixing mining,
the brief will improve scores on monologue-only videos and quietly do
nothing for podcast/interview content — the worst kind of feature
where it works in our demos and fails in user content.

**Updated build sequence:** miner-quality first (multi-speaker
candidate emission), then v1.1 brief on top of the now-richer
candidate set.

### Implementation cost estimate

- New Pydantic model + JSON schema: small.
- New orchestrator stage + handoff request/response files: medium.
- Wiring into `scoring-request.json` payload + scoring prompt update:
  small.
- Tests (unit + e2e): medium.
- SKILL.md docs update for the new handoff: small.
- One additional model handoff in the user-visible flow (latency + token
  cost): real but bounded — one call per video, cached.

Total: ~1 day of focused work, plus dogfood iteration on a handful of
videos to tune the brief prompt.
